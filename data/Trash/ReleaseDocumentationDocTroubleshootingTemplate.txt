%NOTE% The following sections are intended to be a super set for a Troubleshooting Document.  Particular sections may not apply to all documents, but use all the sections that make sense for your document.
%NOTE% Items in common with many troubleshooting documents should have their own page and be pointed to by or included in this document.
%NOTE% Create a new Troubleshooting document if some software or operational change requires more than a few changes to this procedure and the old troubleshooting procedure still needs to be available for use.

---++ Introduction
Brief description of troubleshooting procedure, what it applies to, and how this fits into any larger process.

---++ Common problems
List or describe the most common problems or errors that occur that would have caused the reader to be using this document.
For example:
   1. Do you have a valid proxy?
   1. Have you typed commands correctly?
   1. Are you using the right directory?
   1. Do you have access to the service?
   1. Are you authorized to use the service?
   1. Etc.

---++ Troubleshooting Procedure
The description of the steps to step the reader through in order to eliminate possible problem areas and identify the area that is causing the the problem.   
For each step, you should have what to do, what you should expect to see, and what to do if you don't get the expected response.
Each step that requires unix commands should show the commands in a green box using 
<verbatim><pre class="screen">
command parameters
</pre></verbatim> tag which will look like this.
<pre class="screen">
command parameters
</pre>
%BR%
%NOTE% Standard variables (such as $VDT_LOCATION) should be used where they exist
%NOTE% The terminal interaction must be verbatim with no extra characters so that a user can copy and paste the command lines.

---++ If You Need Help
This should cover things the reader should do in order to get help from experts.

---+++ What information an expert will need
Identify information that anyone assisting with this problem will need and where or how the reader can get that information.

---+++ Who to contact
This should cover the best way to obtain help on this problem.  It could direct the reader to the GOC, the Integration team, a VO support center, the security team, a mailing list, or some other expert group.  Include both who and how to contact them.

---+++ References
References to additional information.  This document should be as concise as possible, but the reader can use these links if he/she wants to obtain more background or information.
