The following fields in the <nop>VomsUserGroup element in _gums.config_ that require a little clarification:
<verbatim>
    acceptProxyWithoutFQAN = yes | no
    matchFQAN = ignore | vo | group | exact
    voGroup
    voRole
</verbatim>

The _voGroup_ and _voRole_ attributes affect the set of members retrieved from VOMS:
%TABLE{ tableborder="1" cellpadding="0" cellspacing="1" headerbg="#99CCCC" databg="#FFFFCC, #FFFFFF"}%
%EDITTABLE{ header="|  |  |  | " format="| textarea, 1x10 | textarea, 1x10 | textarea |  " changerows="on" quietsave="on" editbutton="Edit table" }%
|*voGroup*|*voRole*|*Membership retrieved*|
| not specified | not specified | all members of the VO |
| /cms/uscms | not specified | all members of the specified group in the VO |
| /cms/uscms | cmsprod | all members of the specified group with the specified role in the VO |
| not specified | cmsprod | *INVALID specification* - no members retrieved |

<b>Note: </b> _acceptProxyWithoutFQAN_ and _matchFQAN_ attributes have no affect on member retrieval.

The _acceptProxyWithoutFQAN_ and _matchFQAN_ attributes affect the authorization policy for that set of members (_userGroup_):


matchFQAN=exact  =>  match if user's FQAN equals voGroup/Role=voRole (user's DN not searched for in database)  <br>
matchFQAN=group  =>  match if user's FQAN starts with voGroup (user's DN not searched for in database)  <br>
matchFQAN=vo  =>  match if very first part of user's FQAN (before 1st and 2nd slash) equals first part of voGroup (user's DN not searched for in database)  <br>
matchFQAN=ignore  =>  match if user's DN found in database  <br>

There may be some subtle problems with this logic.  For example, it may be problematic if a VomsUserGroup element is created that specifies
    matchFQAN=ignore
    voGroup=/atlas/usatlas
    voRole=production
Then, a user with DN
    /DC=org/DC=doegrids/OU=People/CN=John Smith 12345
and FQAN
    /atlas/usaltas/Role=software
will never get retrieved from VOMS and will thus never be matched for membership even though you might expect him to be since matchFQAN was set to ignore.  Should this be disallowed?  That is, should it not be allowed to use matchFQAN=ignore with setting voGroup and voRole?  Or this a valid case to allow for?
