*Introduction*

This is the design specification for the OSG Storage Discovery capability. It is written to define how the requirements described in the companion requirements document (see References) shall be implemented.

*System Overview*

The basis for much of the required functionality for OSG Storage Discovery has already been implemented and deployed, in the sense that services which can deliver much if not all of the needed information already exist within the OSG. In particular, the BDII service aggregates storage as well as compute information from OSG sites, using recently deployed updated GIP modules. The OSG Resource Selection Service has also been deployed, aggregating information using the same schema (Glue 1.3) as BDII, through CEMon. The OSG Matchmaker software provides <nop>ReSS information supplemented with additional site snapshots obtained by probes. The OSG Storage Discovery project will provide an accessible means of exploiting the information provided by these services.

The project will ideally not require modifications to services on which it depends, and this specification is written to that end. That being said, it is possible that some shortcomings in the existing services will be exposed by the new capability, so the door should remain open to interplay between the design of this component and those of the above services.

The requirements document describes an Authorization Check feature as being optional. In this document, that feature will be considered as out of scope.

*Design Considerations*

The new software falls in to the Ease-of-Use category targeted at new users of the OSG. Therefore, installation and use of the software should be as simple as possible. Fortunately, most of the needed functionality can be achieved with well-known web service components, and the areas that cannot, can be packaged in such a way as to hide the complexity from the user.

*Architectural Strategies*

This software lies at the consumer end of the information services, providing access through interfaces that most users are already familiar with. Therefore, the architecture will serve as a bridge between those interface components and the existing services.

*System Architecture*

The system architecture for the Information Services domain of the Open Science Grid is described in ??????. The portions of interest to OSG Storage Discovery are the BDII and ReSS services. The new software shall interact with the BDII service through the LDAP interface. It shall interact with the ReSS service indirectly, through the test job run function acting through schedd.

The eventual use of schedd causes this software to also have a dependence on the Job Submission domain, at least to the extent of the required format for job submission materials. There is, however, not effect of this dependence on the software design.

*Software Architecture*

To implement the features described the requirements document, the following specification shall be followed. See figure 1. for an overview of the software architecture.

Satisfying requirement 1.1, the requirements expression shall be written in XPath. This format allows easy composition of multiple parameters, allows the use of commonly-available libraries and representations, and provides for future integration capability.

A command line interface to BDII shall be written in Java, satisfying requirements 2.1-2. The interface shall use openldap libraries for communication with BDII. The XPath expression shall first be converted to XML using the XQueryX utility, and then to ldapsearch format via the STAX parser using custom software. Results from the BDII server shall be converted from DSML to structured XML, and the XPath expression evaluated using the Xalan library.

Command line options shall be as follows.

_Usage_
<verbatim>
xpathsearch [options]  <xpathexpression | name1<OP>value1 [name2<OP>value2] ...>
</verbatim>

_Available options_
<verbatim>
 -b <o=BASE>                       base for LDAP search. Example (default): -b o=grid

 -H <PROTOCOL://HOST:PORT>         host_uri for search. Example (default): -H ldap://iu.grid.ui.edu:2710

 -h,--help                         Print usage and options

 -v,--version                      Print the version number of xpathsearch

 -u,--use_ldap_aux                 Use auxilary LDAP classes

 -d,--output_ldap_dn               Output the dn of each LDAP objectclass

 -a,--output_all_ldap_attributes   Output all queried attributes of each LDAP objectclass

 -t,--test                         Run the unit tests
</verbatim>

_Explanation of Options_
<verbatim>
 -b <o=BASE>                       base for LDAP search. Example (default): -b o=grid
</verbatim>
The base determines at what level in the LDAP directory the search begins. The default "o=grid" is the highest level of the BDII information. A lower level, such as to produce a search on the information from only one site, for example, could be specified by using the site's LDAP dn as the base. 

<verbatim>
 -H <PROTOCOL://HOST:PORT>         host_uri for search. Example (default): -H ldap://iu.grid.ui.edu:2710
</verbatim>
URI of the information server. Only the ldap protocol is supported in the initial implmentation.

<verbatim>
 -h,--help                         Print usage and options
</verbatim>
Print the above usage synopsis and option, and the supported attribute name shortcuts (see below).

<verbatim>
 -v,--version                      Print the version number of xpathsearch
</verbatim>
The version is the release number for production software, and build number for test software.

<verbatim>
 -u,--use_ldap_aux                 Use auxilary LDAP classes
</verbatim>
Standard LDAP searches ignore auxilary classes and utilize structural classes only. This causes inconsistency with the corresponding XML schema, which treats all classes the same. This feature allows switching between the two conventions.

<verbatim>
-d,--output_ldap_dn               Output the dn of each LDAP objectclass
</verbatim>
In cases where the search is on only nodes, nothing useful would be output because all specific information is in attributes. This option specifies a default attribute on which to produce the results.

<verbatim>
 -a,--output_all_ldap_attributes   Output all queried attributes of each LDAP objectclass
</verbatim>
Similar in motivation to the "--output_ldap_dn" option, but outputs all attributes.

<verbatim>
 -t,--test                         Run the unit tests
</verbatim>
Invokes the unit test, which contains some XPath expressions which can be run against the default host and base.

_xpathexpression Format_

In this format, the XPath expression is supplied by the user. It must conform to the schema of the server specified with the "-H" option. All XPath 1.0 expressions are supported, with the following exceptions.
   * Lateral traversal notations, such as "next-sibling", are not supported. Child and ancestor traversal notations are supported.
   * The use of variables is not supported.

_Name/Value Format_

A series of attribute value comparisons may be substituted for the XPath Expression. The names on the command line must correspond to attribute names in the schema. The values are specified by the user, and will be compared to the associated values from the information service trough the "OP", which must be one of "<", "=", or ">" The software will resolve the attribute names to their full paths and compose internally the corresponding XPath expression for the logical AND of the set of attribute value comparisons specified.

Shorthand attribute names may be used as follows:

   * access=GlueSEAccessProtocolType
   * control=GlueSEControlProtocolType
   * vo=GlueVOInfoAccessControlBaseRule
   * lifetime=GlueSAPolicyFileLifeTime

Other shorthand names may be added as the need arises. All the available shorthand names will be printed when the "-help" is chosen.

The user interface shall be written in Javascript, runnable in all Firefox, Safari, and Internet Explorer browsers. Graphical components are as follows. Requirement 1.2 shall be satisfied with a pasteable input text field and a result text field. A tree view of the BDII information shall provide the browsing function for requirement 1.3. A text field will provide XPath snippets from the current selection in the tree view to aid in the construction of XPath expressions. A dropdown menu will provide preset searches for requirement 1.4. A dropdown menu shall provide a variety of forms for requirement 1.5, each being implemented with text fields and radio buttons. Requirements 1.6 and 1.7 shall be provided by text fields. A button with associated output fields shall satisfy requirement 1.8. The outputs defining a runnable test job shall be copyable for pasting into an external editor. If security constraints allow, the text shall also be saveable as a files. In all cases the above components shall either lead to internal javascript processing, or formation of requests to the proxy server, described below.

The test job submission files shall be collected by the user submitted outside of the OSG Storage Discovery application. Therefore the test job submission files must be compatible with the job submission software the user is directed to employ.

A REST proxy web service shall be run, either remotely or locally, connecting the user interface to the Java application. Messages in http format shall be sent from the javascript instance to cause the proxy to invoke methods from the Java command line interface. The proxy service shall respond to the javascript instance using XML over http. As time permits, a caching capability shall be incorporated into the proxy server with XMLDB.

The user interface shall be made available through OSG's MyOSG service. The proxy server shall run in conjunction with that service. As such, it shall be deployed in a manner compatible with Apache 2. The deployment will be in the form of a Sun Grizzly application server utilizing an Apache http connector.

*Risk*

 Conversion of XPath expressions to LDAP search strings is difficult, but has been mostly accomplished in the development so far.

 The degree is difficulty of the javascript code is not completely known, but is not expected to be significantly more than for java swing code, for which the above description would not cause concern.

 Installation of a proxy service and optional cache is an extra step which could cause difficulties for installation and firewall issues with novice users. That could be mitigated by the deployment of a central proxy service by OSG.

 Information gathered in BDII and ReSS could be inaccurate or incomplete, resulting in a failure of the test job to validate the search results.

 Widespread simultaneous use of the tool without caching would cause an increased load on the BDII server.
